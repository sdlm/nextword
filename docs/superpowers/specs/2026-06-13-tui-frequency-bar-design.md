# Замена столбца номеров на бар частотности (wordfreq)

**Дата:** 2026-06-13
**Статус:** утверждён, готов к планированию

## Проблема

В TUI каждая строка слова показывает два номера — глобальный (`global_num`,
он же `id`) и порядковый внутри подуровня (`sublevel_num`) в формате
`{global_num:<5} / {sublevel_num:>3}`. Эти номера занимают ~11 символов и не
несут полезной для пользователя информации при просмотре словаря.

Полезнее видеть, насколько слово частотно в реальном языке. Библиотека
`wordfreq` (уже добавлена в зависимости) даёт Zipf-оценку частотности.

## Цель

Убрать оба номера из отрисовки строки и на их месте показывать **бар
частотности** — заполненность бара пропорциональна Zipf-частотности слова
(полный бар = частотное/обыденное слово, пустой = редкое).

## Ключевое различие двух номеров (не симметричны!)

- **`global_num` (= `w["id"]`) — несущее поле.** Используется вне отрисовки:
  - ключ выбора галочек `_checked_ids` (`app.py:264`);
  - ключ сохранения позиции `_current_word_id` → `_save_position`
    (`app.py:302`).

  Из строки убирается **только отрисовка** этого номера. Поле `global_num`
  в `WordRow` и данные `id` остаются без изменений.

- **`sublevel_num` — чисто отображаемое поле.** Используется только в
  `_text()` (`app.py:86`). Удаляется полностью: и из `WordRow`, и его
  вычисление в `db.py` (`get_words`, `get_all_words`).

Проверка: `grep -n global_num` показывает ссылки вне `_text()`,
`grep -n sublevel_num` — нет.

## Распределение частотности в словаре (обоснование клампа)

Замер по всем 5494 словам БД (`data/words.db`):

```
zipf==0 (нет в wordfreq): 33 слова
min=0.00  max=6.08  mean=3.48  median=3.51
p5=1.87  p10=2.25  p25=2.87  p50=3.51  p75=4.12  p90=4.69  p95=5.02  p100=5.96
```

Линейная шкала Zipf 0–8 дала бы вечно полупустые бары. Поэтому реальный
диапазон словаря клампится к `[2.0, 6.0]` — так бары используют весь размах,
а средние 50% слов дают заполнение ~2..5 из 10 (различимо).

## Дизайн

### 1. `nextword/db.py` — источник частотности

- Импорт `wordfreq` на уровне модуля.
- В `get_words` и `get_all_words` к каждому словарю добавляется поле:

  ```python
  "freq": wordfreq.zipf_frequency(row["word"].strip().lower(), "en"),
  ```

- Удаляется вычисление `sublevel_num`:
  - в `get_words` — поле `"sublevel_num": i + 1` (и больше не нужен `enumerate`);
  - в `get_all_words` — словарь `sublevel_counters` и поле `"sublevel_num"`.
- Поле `id` не меняется.

### 2. `nextword/app.py` — отрисовка

**Хелпер (чистая функция, тестируемая отдельно):**

```python
FREQ_MIN = 2.0
FREQ_MAX = 6.0
FREQ_BAR_WIDTH = 10

def _freq_bar(freq: float, width: int = FREQ_BAR_WIDTH) -> str:
    clamped = max(FREQ_MIN, min(FREQ_MAX, freq))
    filled = round((clamped - FREQ_MIN) / (FREQ_MAX - FREQ_MIN) * width)
    filled = max(0, min(width, filled))
    return "[" + "#" * filled + "-" * (width - filled) + "]"
```

`_freq_bar` возвращает **чистую** (неэкранированную) строку — так юнит-тесты
читаемы. Экранирование делается в месте встраивания (`_text()`), см. ниже.

> **Готча (проверено эмпирически).** Открывающая `[` бара парсится Rich/Textual
> как markup-тег, а `#` — как попытка hex-цвета. Неэкранированный
> `[##########]` **молча исчезает** при рендере (без исключения!) — поэтому
> проверка подстроки в сырой `_text()` НЕ ловит баг. Бар обязан экранироваться
> при встраивании, как уже делается для чекбокса (`\\[{mark}]`).

**`WordRow`:**
- Конструктор: параметр `sublevel_num` удаляется; добавляется `freq: float`.
  Параметр `global_num` **остаётся** (несущее поле).
- `self.freq = freq` сохраняется как атрибут.
- `_text()` — новый формат строки:

  ```python
  from rich.markup import escape  # импорт на уровне модуля app.py
  ...
  line = (
      f"\\[{mark}] {self.level:<3} {sub_short:<6}"
      f"  {escape(_freq_bar(self.freq))}  {self.word:<28} {first_line}"
  )
  ```

  Бар оборачивается в `escape(...)` (иначе молча исчезает — см. готчу выше).
  Цветовая обёртка строки (`[yellow]` для checked, `[green]` для loaded) —
  без изменений. Бар монохромный: вложение цвета в бар конфликтовало бы
  с обёрткой всей строки.

**Места создания `WordRow` (`compose`, `_rebuild_list`):**
- `sublevel_num=w["sublevel_num"]` → удаляется.
- Добавляется `freq=w["freq"]`.
- `global_num=w["id"]` — без изменений.

### 3. Итоговый вид строки

Строки формула-точны (`make` zipf 6.08, `allow` 5.01, `abandon` 4.02,
`quotidian` 2.23):

```
[ ] A2  begin   [##########]  make       делать
[ ] B1  interm  [########--]  allow      позволять
[ ] B2  advanc  [#####-----]  abandon    покидать
[ ] C1  advanc  [#---------]  quotidian  обыденный
```

## Edge-cases

- **Слово не в wordfreq (`freq == 0`)** — 33 слова. Пустой бар
  `[----------]`. Семантически означает «не найдено», а не строго
  «редчайшее»; для UI приемлемо.
- **Многословные фразы** — `zipf_frequency` отдаёт низкое/нулевое значение →
  почти пустой/пустой бар. Приемлемо.

## Тестирование

- **`tests/test_db.py`:** каждая запись `get_words`/`get_all_words` содержит
  ключ `freq` типа `float`; частотное слово (например `make`) даёт значение
  больше, чем редкое; `sublevel_num` больше не присутствует.
- **Новый тест на `_freq_bar`** (граничные случаи, на чистой строке):
  - `freq <= FREQ_MIN` (вкл. 0) → `[----------]` (0 заполнено);
  - `freq >= FREQ_MAX` → `[##########]` (10 заполнено);
  - среднее значение → частичное заполнение, длина строки всегда `width + 2`;
  - кламп: значение ниже `FREQ_MIN` и выше `FREQ_MAX` не ломает границы.
- **Рендер-тест `WordRow._text()`** (обязателен из-за готчи): отрисовать
  результат `_text()` через `rich.console.Console` в строковый буфер и
  убедиться, что бар реально присутствует в **отрисованном** выводе
  (подстрочная проверка сырого `_text()` баг не ловит — бар молча исчезает).
  Также проверить, что номер слова (`global_num`) и разделитель ` / ` в
  выводе отсутствуют.

## Вне scope

- Цветовая раскраска бара по частотности.
- Изменение колонок level / sublevel / word / translation.
- Сортировка или фильтрация по частотности.
- Кэширование/предрасчёт `freq` (замер показал: лукап по всем 5494 словам
  near-instant, отдельная оптимизация не нужна).
